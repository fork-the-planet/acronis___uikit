# Token contract & versioning policy

What consumers may depend on, what counts as a **breaking** token change, and how
a `@acronis-platform/design-tokens` version maps through the generated
`@acronis-platform/tokens-pd` to its consumers. Source of truth for issue #176
(E1). Sits on top of the repo-wide release flow in
[`../../../context/releasing.md`](../../../context/releasing.md) (Changesets) —
this file adds the **token-specific** rules that generic semver guidance can't.

## The version chain

A token change ripples across three published hops. Each hop has its **own**
version and its **own** changeset:

```
@acronis-platform/design-tokens   (DTCG JSON — tiers/*.json, schemas/*)
        │  built by tools/style-dictionary
        ▼
@acronis-platform/tokens-pd       (generated CSS: --ui-* custom properties,
        │                          per-brand + per-component, Tailwind presets, DTCG)
        ▼
consumers                         (@acronis-platform/ui-react and downstream apps)
```

Key consequence: **`tokens-pd` is generated and committed**, and CI gates on
drift. Any `design-tokens` change that alters generated output **must** rebuild
`tokens-pd` (`pnpm --filter @acronis-platform/tokens-pd build`) and ship both
bumps **in the same PR**. A `design-tokens` change that does _not_ alter any
generated byte (e.g. an `$extensions` metadata tweak) bumps `design-tokens` only.

## What is — and isn't — the public contract

Versioning is judged against the **public contract**, not the file internals.

### `tokens-pd` — the real consumer-facing contract

This is what consumers bind to, so this is what versioning protects:

- **`--ui-*` custom-property names** and their **semantic meaning** (e.g.
  `--ui-background-brand-primary` = the primary brand fill). Names + meaning are
  the API.
- **Per-brand entry points** — `css/<brand>.css` and
  `css/<component>/<brand>.css` (the opt-in component tiers consumers `@import`).
- **The brand-override mechanism** — `acronis` is the full base; other brands are
  override-only. See [`brand-matrix.md`](./brand-matrix.md).
- **Tailwind presets** (`tailwind/**`) and the **DTCG output** (`dtcg/**`) shape.

### `design-tokens` — the upstream source contract

Its `exports` expose `tiers/{primitives,semantic,components}.json` and
`schemas/*`. Its primary consumer is the build tool, but because it's published,
a direct JSON consumer is possible. What's contractual here:

- semantic/component token **paths** (`background.brand.primary`), `$type`, and the **`values` mode
  keys** (themes `light`/`dark`; brands `acronis`/`brand-b`/…); primitive palette paths (`palette.*`) are internal (see below).
- the **alias structure** (`components → semantics → primitives`) only insofar as
  it changes a resolved value or a token's existence.

### Not the contract (internal — change freely, cosmetic bump at most)

- **Primitive palette token names** (`palette.blue.7`) and the tier file layout —
  internal plumbing; consumers bind to semantic/component `--ui-*`, not palette.
- **Exact rgb channel values** of a token whose _meaning_ is unchanged — a
  cosmetic recolor (see below).
- The intermediate `dtcg/` build views inside `tools/style-dictionary`.
- `$extensions` bookkeeping (`com.figma.variableId`, …).

## What is a breaking token change

Classify by the **effect on the public contract**, in this order:

| Category                              | `tokens-pd` (consumer-facing)                                                                                                                                                                                                                                                                                                  | `design-tokens` (source)                                                                                                                             |
| ------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Breaking** (structural or semantic) | • **Remove** a `--ui-*` token<br>• **Rename** a `--ui-*` token<br>• **Re-purpose** a token so its meaning flips (e.g. a `*-danger` token now resolves to a non-danger color)<br>• **Remove/rename** a brand or component entry point (`css/<x>.css`)<br>• Change the brand-override mechanism in a way consumers must react to | • Remove or rename a token path / `$type`<br>• Remove a `values` mode key (drop a theme or brand)<br>• Change a `$type` (e.g. `color` → `dimension`) |
| **Additive** (backwards-compatible)   | • **Add** a new `--ui-*` token<br>• **Add** a brand or component entry point<br>• Add a new brand mode (a new `<brand>.css`)                                                                                                                                                                                                   | • Add a token / group<br>• Add a `values` mode key (new theme/brand)<br>• Add an `$extensions` key (schema-coordinated)                              |
| **Cosmetic / fix**                    | • Adjust a token's **value** while its **meaning holds** (recolor within the same role, tune light/dark) — visual diff, no API break<br>• Re-point an alias to fix a wrong value                                                                                                                                               | • Edit a `$value`/alias whose role is unchanged<br>• Metadata-only edits that don't alter generated output                                           |

The line between **breaking-semantic** and **cosmetic** is meaning, not pixels:
re-skinning `--ui-background-brand-primary` from navy to teal is **cosmetic**
(the role is "primary brand fill", still true); making a token that every
consumer reads as "error red" resolve to green is **breaking** even though the
name is identical, because it silently breaks every consumer's intent.

> Adding a **brand** is always additive — a new Figma brand mode →
> `values.<brand>` → a new override-only `<brand>.css`, no existing token
> touched. The pipeline discovers it with no code change
> ([`brand-matrix.md`](./brand-matrix.md)).

## Bump mapping

We are **pre-1.0** on both packages (`design-tokens` 0.x, `tokens-pd` 0.x) and
We are **pre-1.0** on both packages (`design-tokens` 0.x, `tokens-pd` 0.x) and
the token contract is **not yet frozen** — freezing it is a ui-react 1.0
criterion (see [`../../../context/roadmap.md`](../../../context/roadmap.md)). `releasing.md` says to avoid `major` until

| Change category | Changeset bump (pre-1.0)                                          | Changeset bump (1.0+) |
| --------------- | ----------------------------------------------------------------- | --------------------- |
| Breaking        | `minor` (0.x → 0.(x+1).0) — **call it out loudly in the summary** | `major`               |
| Additive        | `patch`                                                           | `minor`               |
| Cosmetic / fix  | `patch`                                                           | `patch`               |

Pre-1.0, `patch` covers both additive and cosmetic — the bump can't distinguish
them, so the **changeset summary prose must**: lead with `Add …` for additive,
`Adjust …`/`Fix …` for cosmetic. At 1.0 we switch to literal semver (the right
column) and freeze the contract.

## How a bump propagates across hops

A single token change produces **a changeset per affected published package**, of
the **same category**:

1. **Author** the change in `design-tokens` `tiers/*.json` → `pnpm --filter @acronis-platform/design-tokens validate`.
   Add a `design-tokens` changeset (category per the table).
2. **Rebuild** `tokens-pd` (`pnpm --filter @acronis-platform/tokens-pd build`);
   commit the regenerated output. If any byte changed, add a `tokens-pd`
   changeset of the **same category**. (No output change → no `tokens-pd`
   changeset; the drift gate confirms it.)
3. **Consumers** (`ui-react`): a **breaking** `tokens-pd` change means ui-react
   must update the affected `--ui-*` references / `@theme` bridge in the same or
   a follow-up PR, with its own changeset (`minor` pre-1.0). **Additive** and
   **cosmetic** changes need no ui-react code change — `@theme inline` re-reads
   `var(--ui-*)` at paint time, so values flow through with no rebuild; bump
   ui-react only if you want to advertise the new tokens.

Rule of thumb: the bump category is set **once** by the effect on the
consumer-facing `--ui-*` surface, then applied to every published hop the change
touches.

## Authoring checklist

- [ ] Classify the change: breaking / additive / cosmetic (by **contract
      effect**, not file diff size).
- [ ] `pnpm --filter @acronis-platform/design-tokens validate` clean.
- [ ] `tokens-pd` rebuilt; regenerated output committed (or confirmed
      byte-identical).
- [ ] A changeset for **each** affected published package, same category, summary
      prose matching the [bump mapping](#bump-mapping) prefixes.
- [ ] Breaking change: migration note in the changeset summary (old → new name,
      or the meaning that flipped).
